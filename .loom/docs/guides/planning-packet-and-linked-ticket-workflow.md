# Planning packet and linked-ticket workflow

## Use plans when execution strategy deserves its own layer

A plan is the right tool when one bounded body of work needs a durable execution narrative that is broader than a ticket but more concrete than an initiative.

## Build the packet from the right context

A planning packet should pull in the surrounding context that will matter later:

- constitutional direction
- research and evidence
- initiatives when strategy spans multiple downstream artifacts
- specs when behavior is already bounded
- tickets, critique, and docs context where that changes execution strategy

## Keep linked tickets authoritative

Plans own ticket membership, role, and sequencing context.

Tickets still own:

- live status
- execution details
- checkpoints and journals
- verification evidence
- blockers and dependencies

Do not let plans become a shadow ticket ledger.

## Prefer complete execution slices

If the ticket set is already clear, create or update the plan together with linked tickets. If it is not, create the plan first and add tickets once the execution slice is better defined.

## Practical rule

A plan is healthy when it makes the next bounded execution pass easier without forcing readers to reconstruct strategy from a pile of ticket diffs.
