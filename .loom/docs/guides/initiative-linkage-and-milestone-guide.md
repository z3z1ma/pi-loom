# Initiative linkage and milestone guide

## Use initiatives for strategic bodies of work

Create or update an initiative when work spans multiple specs, multiple ticket streams, or a broader strategic objective.

Initiatives should answer why the body of work matters, not just how one unit will execute.

## Keep milestones strategic

Milestones should represent meaningful strategic checkpoints, not tiny task fragments.

A milestone is useful when it helps a future reader understand sequence, dependency, or risk across the broader objective.

## Link the right downstream work

An initiative should stay connected to the work it governs.

Link it to:

- research that supports the strategy
- specs that define intended behavior
- tickets that carry live execution truth
- roadmap items when the initiative advances constitutional direction

## Avoid pseudo-plan behavior

Do not let initiatives become ticket-by-ticket execution journals.

If the content mostly describes sequencing, implementation order, or live execution status, move that detail into plans and tickets instead.

## Practical rule

An initiative is healthy when a reader can understand the objective, risks, milestones, and downstream work without mistaking it for either a spec or a backlog board.
