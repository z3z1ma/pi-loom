## Overview
Pi Loom supports repository-visible `.loom/<module>` markdown projections for selected canonical Loom records so humans can inspect and edit the interactive authoring side of the system without displacing SQLite/Postgres as canonical truth. Projections are revision-aware derived surfaces that reconcile back into canonical records through explicit import/reconcile flows, while packets remain bounded autonomous execution handoffs rather than human editing surfaces.

## Clarifications
- 2026-03-26T07:23:10.881Z Which Loom families receive repository-visible workspace projections, and which remain canonical-only? -> Repository-visible workspace projections cover constitution, specs, plans, initiatives, research, docs, and tickets. Critique and Ralph do not project into `.loom/` because their packet/run artifacts are operational handoffs rather than useful long-lived repository reading/editing surfaces.
- 2026-03-26T07:23:15.798Z What repository layout and metadata shape do workspace projections use? -> Workspace projections live under `.loom/<module>/...` with many markdown files per module family plus one manifest file per family. Pi Loom must not create a per-markdown JSON sidecar. The manifest carries projection metadata, canonical refs, revision tokens, and path indexing for that module family.
- 2026-03-26T07:23:21.301Z How should workspace projections minimize repository churn and Git noise? -> Projection renderers and manifests must omit volatile fields that do not change human-meaningful content, including routine updated-at timestamps when they are not semantically required. Pi Loom should write projection files only when rendered content or stable metadata actually changes, preserve deterministic ordering, and create `.loom/.gitignore` with safe defaults for ephemeral projection internals and high-churn families so repositories do not accumulate avoidable diff noise.
- 2026-03-26T07:23:32.721Z When do exports and imports occur inside the current Pi-centered harness lifecycle? -> In v1, canonical-to-disk export is automatic only after successful canonical mutations performed through Pi Loom commands or tools for enabled projection families, with writes coalesced so only changed projections are rewritten. Disk-to-canonical import never happens implicitly on file save; it occurs only through explicit reconcile/import actions. On session_start, Pi Loom may ensure projection directories and `.loom/.gitignore` exist and scan manifests for dirty projections, but it must not auto-import local edits. Before autonomous packet-based execution or any canonical mutation that depends on a projected record, Pi Loom must fail closed or require reconcile when the relevant projection is dirty so bounded execution always runs from canonical truth rather than stale disk edits.
- 2026-03-26T07:23:38.573Z How should ticket projections balance usefulness with long-term repository sprawl and churn? -> Ticket projections are intentionally bounded. By default, Pi Loom exports open tickets, tickets linked from active plans, and recently updated tickets within a configurable recency window; older closed tickets remain canonical-only and may be pruned from `.loom/tickets/` without affecting canonical history. `.loom/.gitignore` should ignore `.loom/tickets/` by default so ticket projections stay available for local inspection without forcing high-churn execution history into Git, while advanced users may opt in to tracking that directory.
- 2026-03-26T07:23:44.365Z How should plan projections reflect their time-bound nature without creating unnecessary rename churn? -> Plan projection paths are date-oriented using an immutable created-date segment chosen at first export. The date becomes part of the plan file path and remains stable for that plan thereafter, so plans read as time-bound artifacts without being renamed every time they are updated.
