## Overview
Pi Loom must support a first-class operating mode where one Loom space spans multiple repositories that together form a coherent system. In this mode, a human or AI may start Pi from a parent directory above several repositories or from inside any participating repository and still discover, create, read, update, and orchestrate constitution, research, initiatives, specifications, plans, tickets, critique records, Ralph runs, and documentation across the shared system without ambiguity, clone-local leakage, or cwd-dependent misrouting. The system must preserve explicit repository and worktree identity, require truthful scope selection whenever ambiguity exists, keep canonical shared truth portable across machines and future backends, maintain headless parity with human UX, and provide robust diagnostics, migration behavior, runtime propagation, export/sync semantics, and verification guarantees strong enough for production use.

## Clarifications
- 2026-03-22T22:34:11.315Z How must sync/export/import behave for a multi-repository space? -> Export and import semantics must be truthful about scope. A full-space export includes all repositories, worktrees, canonical entities, links, events, and relevant runtime attachments permitted by policy for that space. A repository-scoped export is allowed only when explicitly requested and must be labeled as partial. The system must not implicitly export only the current cwd repository while presenting the result as a complete space snapshot. Hydration/import must preserve multi-repository relationships and detect conflicts without collapsing repository identity.
- 2026-03-22T22:34:28.501Z What is the canonical meaning of a multi-repository Loom space? -> A multi-repository Loom space is a first-class shared coordination boundary that may contain zero or more repositories and zero or more worktrees per repository. It is not inferred to be identical to one repository root, one working directory, or one machine-local clone. A repository belongs to exactly one canonical space at a time, while multiple local worktrees or clones may attach to that repository inside the same space. Starting Pi from a parent directory may help select a space, but the selected space remains an explicit canonical object with stable identity independent of that directory path.
- 2026-03-22T22:34:43.052Z How should Pi behave when a parent directory contains multiple repositories or multiple candidate spaces? -> The system must never silently guess a wrong coordination boundary. When cwd alone does not identify one unambiguous target, Pi must surface deterministic discovery results and require explicit space or repository selection before any repo-sensitive operation executes. Once selected, that binding may be persisted for convenience, but persisted bindings must be inspectable, revocable, and safe to ignore when stale. Safety beats convenience: ambiguous context must produce a truthful prompt or error, not a plausible synthetic repository identity.
- 2026-03-22T22:34:50.619Z What execution scope must runtime launches and path-bearing operations use in a multi-repository space? -> Every runtime launch, subprocess, verifier action, file operation, and path-bearing argument must execute against an explicit repository and, when relevant, an explicit worktree. A parent space provides shared discovery and shared canonical context; it does not license implicit cwd-based execution against an arbitrary child repository. Bare relative paths are valid only when the active repository scope is explicit and unambiguous. Multi-repository sessions must propagate scope metadata across Ralph, critique, docs, and any nested harness session so downstream work cannot silently drift to the wrong repository.
- 2026-03-22T22:35:01.033Z What information must remain canonical shared truth versus clone-local runtime state? -> Canonical records must remain portable and intelligible without any specific clone, absolute path, local process id, or machine-specific filesystem layout. Space membership, repository identity, repository-qualified refs, cross-entity links, decisions, plans, tickets, and accepted outcomes are canonical shared truth. Worktree leases, local process handles, temporary launch descriptors, cache directories, and other machine-local execution details remain clone-local attachments. Cross-repository support must strengthen this boundary rather than blur it.
