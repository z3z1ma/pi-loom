## Summary
Implement structured checkpoints, compact telemetry, heartbeat freshness, and dashboard/packet views so managers can supervise workers from concise durable state instead of replaying full transcripts.

## Context
Why this exists: the worker substrate needs observability that is rich enough for supervision but compact enough to scale. This ticket creates the core visibility layer that the manager will later use for decision-making. It is directly inspired by the `pi-supervisor` insight that compact state plus recent deltas is more useful than transcript sprawl for long-horizon oversight.

What a newcomer should understand before touching code:
- Checkpoints are richer than heartbeats. A checkpoint summarizes understanding, recent work, validation state, blockers, and next intended action.
- Telemetry/heartbeats are lightweight and primarily answer: is the worker busy, idle, blocked, waiting for review, stale, or finished?
- Dashboards/packets should let a fresh manager session quickly see who needs attention and why.
- This ticket is about observability and durable summaries, not yet about the manager’s final intervention policy.

Likely files and areas touched:
- checkpoint and telemetry domain models/store helpers
- dashboard and packet rendering for workers
- tool/command handlers for appending checkpoints and reading summaries
- tests for freshness, compactness, and restart-safe reads

Important design boundaries:
- Do not depend on live UI for correctness; durable records are canonical.
- Keep checkpoint schema structured and concise; avoid turning it into a transcript dump.
- Preserve enough detail for a new session to resume supervision without opening the entire worker chat.

Dependencies and sequencing:
- Depends on scaffold, worker model, and messaging tickets.
- Unblocks supervision and approval tickets because managers need checkpoint and freshness state to decide what to do next.

## Plan
1. Define checkpoint and telemetry record shapes, including freshness timestamps and high-level status fields.
2. Implement append-only checkpoint writes and lightweight heartbeat/telemetry updates.
3. Generate worker dashboard and packet summaries that foreground stale workers, blockers, pending approvals, and consolidation readiness.
4. Expose operator/agent surfaces to record checkpoints and inspect worker summaries.
5. Add tests for checkpoint persistence, stale-heartbeat detection, and summary rendering quality.
6. Keep the schema compact and durable so a future manager runtime can consume it cheaply.
