## Summary
Build the manager-owned consolidation flow that takes approved worker output and explicitly fans it into the target branch or execution stream while recording strategy, validation, conflicts, rollback needs, and final outcome as durable evidence.

## Context
Why this exists: fan-in is where parallel work stops being embarrassingly parallel. The manager-worker architecture needs a truthful consolidation stage because this is the point where branch conflicts, validation failures, and sequencing decisions become real. This ticket implements the single accountable consolidation point rather than leaving merge behavior implicit or ad hoc.

What a newcomer should understand before touching code:
- Consolidation happens only after explicit manager approval from the prior ticket.
- The manager owns the consolidation decision and records the chosen strategy.
- Strategies may include merge, cherry-pick, patch application, or similar explicit mechanisms; the implementation should not hard-code an unreviewable 'magic merge' path.
- Validation after fan-in matters just as much as the worker’s pre-approval evidence.
- Tickets remain the live execution ledger even though worker state records consolidation details.

Likely files and areas touched:
- consolidation outcome models/store helpers
- runtime/ops helpers for applying approved worker changes
- ticket update hooks for feeding approval/consolidation outcomes back into ticket truth
- tests around conflict handling and failed validation after fan-in

Important edge cases:
- merge conflicts during fan-in
- post-consolidation validation failure
- rollback/follow-up required after partial application
- multiple approved workers converging on the same target branch in sequence
- manager needs to defer one approved worker while consolidating another first

Dependencies and sequencing:
- Depends on runtime attachment and approval tickets.
- Feeds provenance integration because consolidated outcomes must be visible to tickets/plans/Ralph later.

## Plan
1. Define durable consolidation outcome records, including chosen strategy, validation result, conflict state, and rollback/follow-up fields.
2. Implement manager-facing operations for applying approved worker changes into the target branch or execution stream.
3. Record success and failure outcomes durably rather than collapsing them into transient chat text.
4. Update ticket history with key approval and consolidation results while keeping worker records as complementary execution detail.
5. Add tests for successful fan-in, merge conflicts, failed validation, rollback-needed paths, and deferred consolidation sequencing.
