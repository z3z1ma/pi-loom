## Summary
Refactor worker execution behind a runtime abstraction so the current subprocess runner becomes one strategy rather than the architecture itself, making room for SDK-backed and RPC-backed live workers without duplicating worker domain semantics.

## Context
Why this ticket exists: the current worker package still treats one-shot subprocess execution as the dominant runtime assumption. That is acceptable for the first substrate, but it now blocks the next phase. Before adding SDK-backed live workers, the package needs a runtime interface that cleanly separates worker domain semantics from transport/execution details.

What a newcomer should understand:
- this ticket is not yet 'build live workers'; it is 'stop subprocess from being the only shape the code can think in'
- worker semantics must stay the same across runtime kinds: inbox processing, checkpoints, telemetry, approvals, and recovery cannot drift by runtime implementation
- runtime kind should become explicit and durable for debugging and recovery

Likely files / areas:
- `packages/pi-workers/extensions/domain/runtime.ts`
- `packages/pi-workers/extensions/domain/store.ts`
- worker models/descriptors for runtime kind metadata
- worker launch/resume tool handlers
- runtime-focused tests

Important constraints:
- subprocess runtime must continue working as the compatibility baseline
- do not duplicate worker domain logic per runtime kind
- preserve portable canonical state; runtime-specific machine-local details still belong in runtime descriptors

This ticket depends on inbox-driven turn semantics because the runtime abstraction should host a stronger worker contract, not lock in today’s weaker behavior.

## Plan
1. Define a worker runtime interface or equivalent abstraction layer.
2. Move current subprocess execution behind that interface.
3. Persist runtime kind and related execution metadata durably.
4. Ensure launch/resume/recovery paths use the abstraction rather than hard-coding subprocess logic.
5. Add tests proving subprocess behavior still works and runtime kind is visible without changing worker semantics.

## Notes
Architecture seam ticket. Make runtime strategy pluggable before building live workers.
