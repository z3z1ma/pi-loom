## Summary
Close the implementation slice with comprehensive failure-aware tests and package/operator documentation so the new worker substrate is both defended by automation and understandable to future maintainers with minimal prior context.

## Context
Why this exists: the worker substrate is a long-horizon control surface. If it ships with shallow tests or vague docs, the first future maintainer will unknowingly break critical behavior. This ticket makes the design durable by encoding the contract in tests and documentation. It is intentionally last because it should validate the real integrated system rather than stubs.

What a newcomer should understand before touching code:
- This ticket is not a perfunctory cleanup pass. It is where the final contract becomes executable and teachable.
- Tests should cover portability, lifecycle transitions, runtime attachment, messaging, telemetry, supervision, approval, consolidation, provenance, and recovery failure modes.
- Documentation should explain the architecture, boundaries, and operator workflow in repo-truthful terms.
- Prompt guidance tests are part of product behavior and should be included alongside runtime/store/tool tests.

Likely files and areas touched:
- package test suite across `packages/pi-workers/__tests__/`
- `packages/pi-workers/README.md`
- prompt guidance tests and possibly root README references if the public stack description changes
- any missing fixtures/helpers needed for deterministic runtime and store tests

Failure modes that must not be skipped:
- clone-local absolute path leakage into canonical state
- stale heartbeats and repeated blockers
- lost runtime attachments and reprovisioning
- approval without sufficient evidence
- merge/conflict and post-fan-in validation failures
- provenance drift where workers start competing with tickets/Ralph for canonical truth

Dependencies and sequencing:
- Depends on all major behavioral tickets because it validates the finished integrated substrate.
- This ticket is the final gate before treating the manager-worker substrate as an implementation-ready rollout.

## Plan
1. Audit the integrated worker package for every spec capability and ensure there is explicit automated coverage for each critical behavior and failure mode.
2. Add or deepen package tests for portability, lifecycle, launch/resume, messaging, telemetry, supervision, approval, consolidation, provenance, and recovery.
3. Update package README and any public-facing docs/guidance so a novice can understand what workers are, what managers do, and how the substrate fits with tickets/plans/Ralph/critique/docs.
4. Ensure prompt-guidance tests and operator-facing examples encode the same boundaries as the finalized spec.
5. Capture recommended verification commands so future work on the package has an obvious regression suite.

## Notes
Final hardening ticket. Treat this as the contract-locking gate, not as optional polish.
