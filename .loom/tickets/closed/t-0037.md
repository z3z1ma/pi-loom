## Summary
Implement the manager polling loop that scans worker state, unresolved inbox backlog, telemetry, and approvals, then makes durable bounded decisions to message, resume, escalate, or surface approvals without requiring a human between every worker turn.

## Context
Why this ticket exists: once the manager surface exists and live workers are possible, the next useful step is a bounded scheduler loop that reduces manual babysitting. This is the first concrete move toward human-out-of-the-loop usefulness, but it must remain bounded, auditable, and subordinate to ticket/worker truth.

What a newcomer should understand:
- this is not a sidecar mesh or unrestricted autonomous workflow engine
- the scheduler is polling-driven and bounded in this phase
- scheduler decisions must remain durable and audit-friendly
- the manager still owns approvals and consolidation boundaries

Likely files / areas:
- manager command/tool implementation
- worker store/supervision logic
- scheduler runtime/helpers
- manager-focused tests and queue-state rendering

Key responsibilities for the scheduler:
- scan workers and queue state
- inspect unresolved inbox and telemetry
- decide when to message, resume, escalate, or surface approvals
- avoid starving idle workers or endlessly polling with no effect
- respect worker busy/idle/blocked/review states

Important constraints:
- do not create a hidden execution ledger in scheduler memory
- do not automatically bypass manager authority over approval and consolidation
- preserve anti-stagnation behavior rather than replacing it with silent retries
- keep sidecar/event-mesh ambitions out of this ticket

This ticket depends on both the manager surface and the preferred live-worker runtime because the scheduler needs something concrete to drive.

## Plan
1. Design the bounded manager scheduler loop over durable worker state.
2. Implement polling logic that inspects inbox backlog, telemetry, and approval state across workers.
3. Persist scheduler-driven actions durably back into worker/ticket-visible state.
4. Add bounded policies to avoid infinite no-op polling and to escalate stagnation explicitly.
5. Add tests for multi-worker useful progress, escalation on backlog/stagnation, and preserved manager authority over approvals/consolidation.

## Notes
This is the first bounded automation loop for the manager. Keep it useful and auditable, not magical.

## Verification

- manager approval/consolidation authority remains intact even when the scheduler is active
