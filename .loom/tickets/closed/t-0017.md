## Summary
Add the workspace lifecycle and runtime attachment layer that turns a durable worker record into a real Git-worktree-backed execution environment with subprocess-backed Pi launches, while keeping machine-local attachment facts outside canonical committed state.

## Context
Why this exists: the spec says a worker is an abstraction over an ephemeral workspace, typically a Git worktree. This ticket is where that becomes operationally true. Without it, workers would still be durable metadata with no actual isolated execution environment. The main architectural constraint is that current Pi session operations are not workspace operations, so this ticket must not fake workspace isolation by mutating session cwd or relying on session forks.

What a newcomer should understand before touching code:
- v1 explicitly prefers Git worktrees as the backing strategy.
- The durable worker record stores logical workspace intent; the runtime attachment layer materializes that intent on a particular machine.
- Clone-local details such as actual worktree path, process id, or host attachment status belong in runtime-only descriptors.
- The expected runtime form is a subprocess-backed Pi instance launched with cwd set to the provisioned workspace.
- This ticket should make provisioning, attachment, launch, and resume possible without yet implementing the full messaging/supervision surface.

Likely files and areas touched:
- worker runtime/launch domain files, likely `packages/pi-workers/extensions/domain/runtime.ts`
- worker path/helpers and launch descriptor handling
- possibly command/tool handlers for attach/launch/resume prerequisites
- tests that exercise provisioning/attachment behavior in a controlled filesystem fixture

Key edge cases to handle:
- worker record exists before a live workspace has been provisioned
- workspace must be reprovisioned after local cleanup while the worker identity remains stable
- launch descriptor exists but underlying worktree path no longer exists
- branch naming/base ref intent must remain explicit and reconstructible
- runtime attachment must never back-write clone-local paths into canonical state

Non-goals:
- do not build the manager supervision engine here
- do not finalize approval/consolidation semantics here
- do not pretend a missing worktree can be silently tolerated without explicit lifecycle state updates

Dependencies and sequencing:
- Depends on scaffold and worker model tickets.
- Unblocks operator surface, consolidation, and recovery flows.

## Plan
1. Define runtime-only launch/attachment descriptor shapes for worker provisioning and subprocess launch.
2. Implement Git-worktree-oriented provisioning helpers that map the portable workspace descriptor into a local attachment.
3. Add lifecycle transitions for requested -> provisioning -> ready -> active and related failure states.
4. Implement subprocess-backed launch/resume behavior that starts Pi with cwd set to the provisioned workspace.
5. Keep runtime-only descriptors separate from canonical state writes.
6. Add tests for initial provisioning, reprovisioning, stale attachment recovery, and launch/resume reconstruction.
7. Make failures explicit in lifecycle state rather than hiding them in logs or chat-only messages.

## Notes
This ticket is where the substrate stops being conceptual and becomes a truthful workspace-backed runtime.

## Verification
Expected verification for completion:
- A worker can move from logical intent to a materialized local workspace attachment.
- Launch and resume use subprocess-backed Pi execution rooted in the workspace path.
- Runtime-only descriptors can be discarded/recreated without corrupting canonical state.
- Tests cover missing worktree recovery and prove session fork/tree are not used as the primary isolation mechanism.

- `npx vitest run packages/pi-workers/__tests__/runtime.test.ts`
