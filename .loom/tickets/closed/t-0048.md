## Summary
Define and implement the boundary between canonical shared worker history and clone-local worktree/runtime control-plane state, including leases and heartbeats for any canonical worker liveness state.

## Context
Projected from finalized spec sqlite-first-canonical-storage-substrate: SQLite-first canonical storage substrate

Capabilities:
- cap-worker-runtime-carveout: Execution-local runtime carve-out and durable worker history boundary

## Plan
Spec change: sqlite-first-canonical-storage-substrate

Task summary: Define and implement the boundary between canonical shared worker history and clone-local worktree/runtime control-plane state, including leases and heartbeats for any canonical worker liveness state.

Requirements:
- req-021: Durable worker messages, checkpoints, approvals, and completion history may move into canonical storage only if lease/heartbeat semantics prevent stale running state.
- req-022: Sync exports exclude clone-local runtime attachments and process metadata.
- req-023: Worker histories in canonical storage distinguish durable shared facts from machine-local execution details.
- req-024: Worktree paths, PIDs, transient launch descriptors, local runtime attachments, and similar clone-specific state remain local-only.

## Verification
Implemented clone-local runtime attachment storage and lease helpers in `packages/pi-storage/storage/runtime.ts`. The new `LocalRuntimeAttachmentStore` persists runtime attachments under the user-level Loom root but outside the canonical SQLite catalog path, and explicit helpers distinguish stale vs active leases and renew heartbeats without mutating prior records. Verified with `npx biome check packages/pi-storage`, `npm test -- --run packages/pi-storage/__tests__/contract.test.ts packages/pi-storage/__tests__/catalog.test.ts packages/pi-storage/__tests__/sync.test.ts packages/pi-storage/__tests__/runtime.test.ts`, and `npm run typecheck`.
