## Summary
Cut `pi-specs`, `pi-plans`, and `pi-ticketing` over to the shared storage contract while keeping the main spec markdown bodies and selected review projections repo-materialized.

## Context
Follow-up to the foundational storage substrate slice (`t-0044` through `t-0049`). These packages currently drive most cross-layer execution semantics and still treat file-backed `.loom` artifacts as canonical operational truth. This ticket migrates that middle/lower execution stack onto shared storage after the upstream identity/provenance layers are ready.

In scope:
- `packages/pi-specs/`
- `packages/pi-plans/`
- `packages/pi-ticketing/`
- integration with `@pi-loom/pi-storage`

Important carve-out:
- keep `proposal.md`, `design.md`, `tasks.md`, and related human-facing spec text repo-materialized and grep-friendly
- keep selected plan/ticket review surfaces as deliberate projections where useful
- move machine metadata/state and canonical execution graph state into shared storage

## Plan
1. Introduce storage adapters for specs, plans, and tickets on top of `@pi-loom/pi-storage`.
2. Preserve repo-materialized spec markdown bodies and selected review projections while moving canonical machine state to shared storage.
3. Replace direct file-based graph and projection assumptions with shared storage queries and explicit projections.
4. Update projection logic, ticket/spec/plan linkage, and focused tests to reflect the new canonical source of truth.
5. Keep the cutover truthful by removing stale canonical-file assumptions once the shared storage path is active.
