## Summary
SDK-backed worker launches can return `completed` while leaving the worker inbox untouched and producing no checkpoints, findings, or durable progress. This makes the preferred in-process worker path look successful when it has actually done nothing.

## Context
Observed while trying to run `workspace-package-audit-worker` for ticket `t-0040` in the main Pi Loom workspace. The worker was originally launched through the wrong manager-side path and with the wrong runtime (`subprocess`), which I corrected. I then used the worker-specific launch path with the SDK runtime, as intended by the current worker design.

Concrete observed behavior:
- `worker_resume { ref: "workspace-package-audit-worker", runtime: "sdk" }` returned `Execution: completed`
- `worker_launch { ref: "workspace-package-audit-worker", runtime: "sdk" }` also returned `Execution: completed`
- durable worker state now records `lastRuntimeKind: sdk`
- despite that, the assignment message `workspace-package-audit-worker-msg-0001` remains `pending`
- worker dashboard still shows `ready/idle`, unresolved inbox `1`, checkpoints `0`, findings `0`
- no durable worker-side acknowledgment, resolution, checkpoint, or completion request was written

This is not just a manager-flow problem. The preferred SDK runtime path itself is reporting success without evidence that the worker processed its inbox or performed any work. That is a correctness bug because the control plane now lies about execution progress.

Relevant code to inspect:
- `packages/pi-workers/extensions/tools/worker.ts` (`worker_launch`, `worker_resume`)
- `packages/pi-workers/extensions/domain/runtime.ts` (`runSdkWorkerLaunch`)
- `packages/pi-workers/extensions/domain/store.ts` (`startLaunchExecution`, `finishLaunchExecution`, `executionSummary`)

Why this matters:
- SDK-backed workers are the preferred live-worker direction in the package README and architecture guidance
- false-positive completion means managers cannot trust worker launches
- downstream ticket/worker state becomes misleading because success is recorded without any durable action

## Plan
1. Reproduce the no-op SDK launch in a focused test or harness invocation using a worker with a pending inbox assignment.
2. Identify whether the issue is in session creation, prompt execution, event subscription/output handling, or the success criteria used when finishing launches.
3. Change the runtime/store contract so SDK launches are not marked as successfully completed when no durable worker action occurred and the inbox remains untouched.
4. Add regression coverage proving that an SDK launch either produces durable worker progress or fails/blocks truthfully.
5. Re-run the affected worker flow and confirm the audit worker can actually start processing work.
