## Summary
Build the SQLite adapter behind the storage contract, import current file-backed state, move metadata/state into the DB, and generate deterministic repo projections that preserve main constitution/docs/spec markdown bodies for grep/review during the cutover period.

## Context
Projected from finalized spec `sqlite-first-canonical-storage-substrate`. This ticket owns the operational cutover. The important repo carve-out is selective: keep the main human-facing markdown bodies materialized in-repo, but move machine-oriented metadata/state into the DB by default. That means `pi-constitution` should keep the constitutional brief and main constitutional markdown bodies in the repo while moving JSON metadata/state into SQLite, `pi-docs` should keep main doc markdown body/bodies in the repo while moving JSON metadata/state into SQLite, and `pi-specs` should keep `proposal.md`, `design.md`, `tasks.md`, and related human-facing spec text in the repo while moving machine metadata/state into SQLite unless a specific projection is justified.

## Plan
1. Implement the SQLite backend behind the storage contract and import current file-backed operational state.
2. Define deterministic projection generation for repo-facing artifacts.
3. Preserve main constitution/docs/spec markdown bodies in the repo as grep-friendly materialized outputs or editing surfaces.
4. Move constitution/docs/spec JSON metadata, dashboards, machine state, and similar auxiliary artifacts into the DB by default.
5. Prove round-trip import/projection behavior without leaking clone-local absolute paths.
