## Summary
Define and implement deterministic sync exports, content-addressed artifact manifests, hydration/import flows, and explicit conflict handling for SQLite users.

## Context
Projected from finalized spec `sqlite-first-canonical-storage-substrate`. This ticket defines the reserved-branch sync surface for SQLite users. The branch should carry deterministic exports for canonical DB state plus any repo-materialized main constitution/docs/spec markdown bodies or artifact manifests needed for hydration, but it must not treat auxiliary JSON metadata for constitution/docs/specs or raw SQLite files as the merge unit. The intended repo carve-out remains selective: main constitution/docs/spec markdown bodies stay grep-friendly in repos; machine metadata/state lives in the DB unless specifically projected.

## Plan
1. Define a deterministic export format for canonical DB entities, links, versions, and ownership metadata.
2. Define how repo-materialized main constitution/docs/spec markdown bodies and artifact manifests participate in hydration without making auxiliary JSON metadata the merge unit.
3. Implement idempotent hydration/import plus explicit conflict detection.
4. Prove that a fresh machine can reconstruct the same logical graph using the sync branch and repo-materialized markdown/artifact surfaces.

## Verification
Implemented deterministic sync-bundle export and hydration in `packages/pi-storage/storage/sync.ts`. The bundle exports text/entity JSON plus repo-materialized markdown bodies under `repo-materialized/`, and explicitly does not use raw SQLite files or auxiliary docs/spec metadata as the merge unit. Hydration replays spaces, repositories, entities, links, events, and projections into a fresh catalog with conflict detection on entity/projection versions. Verified with `npx biome check packages/pi-storage`, `npm test -- --run packages/pi-storage/__tests__/contract.test.ts packages/pi-storage/__tests__/catalog.test.ts packages/pi-storage/__tests__/sync.test.ts`, and `npm run typecheck`.
