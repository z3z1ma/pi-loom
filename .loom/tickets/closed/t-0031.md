## Summary
Turn the existing worker message stream into a real durable inbox contract so manager instructions, worker acknowledgments, resolutions, escalations, and bounded broadcast state are all queryable, actionable, and impossible to silently ignore.

## Context
Why this ticket exists: the current worker substrate persists messages, but they still behave too much like durable notes. That is not strong enough for useful autonomous manager-worker behavior. Before runtime changes or manager scheduling can become trustworthy, the system needs a formal inbox contract. This ticket is the control-plane foundation for the whole next phase.

What must be true when this ticket is done:
- manager-originated instructions are first-class inbox items
- workers can acknowledge, resolve, or escalate those instructions durably
- the unresolved inbox backlog is directly inspectable from worker reads/dashboards/tools
- there is no need to reconstruct message intent from freeform transcript text

Likely files / areas:
- `packages/pi-workers/extensions/domain/models.ts`
- `packages/pi-workers/extensions/domain/store.ts`
- `packages/pi-workers/extensions/domain/render.ts`
- `packages/pi-workers/extensions/domain/dashboard.ts`
- `packages/pi-workers/extensions/tools/worker.ts`
- `packages/pi-workers/__tests__/store.test.ts`
- `packages/pi-workers/__tests__/tools.test.ts`

Important design constraints:
- preserve canonical worker portability; do not introduce machine-local state into canonical message records
- keep tickets as the live execution ledger; inbox semantics belong to workers, not to ticket state itself
- do not jump to real-time daemons or sidecars here; this ticket is about protocol truth first
- bounded broadcast remains allowed, but unrestricted worker-to-worker chat remains out of scope

Key design questions to answer in code:
- what message kinds exist and which are actionable instructions versus observations?
- how do `pending`, `acknowledged`, and `resolved` transitions work?
- who records those transitions and where is the evidence stored?
- how does a manager query unresolved inbox state cheaply and deterministically?

## Plan
1. Extend worker message models to distinguish manager instructions, acknowledgments, resolutions, escalations, and bounded broadcast notices.
2. Add durable lifecycle state transitions and any associated metadata needed to show who changed a message state and when.
3. Update store helpers so unresolved inbox state is queryable directly rather than inferred indirectly.
4. Surface unresolved inbox counts and recent unresolved items in worker reads/dashboards/renders.
5. Update `worker_*` surfaces so they can write and inspect the stronger inbox state.
6. Add focused tests for lifecycle transitions, unresolved backlog queries, and durable acknowledgment/resolution semantics.

## Notes
Protocol-first ticket. Do not let runtime work outrun explicit inbox semantics.
