## Summary
Add stable global IDs, ownership/project metadata, and version semantics to all durable entity types while preserving human-friendly local refs as aliases or projections.

## Context
Projected from finalized spec sqlite-first-canonical-storage-substrate: SQLite-first canonical storage substrate

Capabilities:
- cap-sqlite-backend: SQLite-first local backend and migration mode
- cap-storage-topology: Shared catalog topology and global identity

## Plan
Spec change: sqlite-first-canonical-storage-substrate

Task summary: Add stable global IDs, ownership/project metadata, and version semantics to all durable entity types while preserving human-friendly local refs as aliases or projections.

Requirements:
- req-001: Every durable entity has a stable app-generated global ID; human-friendly refs remain secondary aliases or projections.
- req-002: Logical coordination containers are modeled explicitly so tickets, plans, initiatives, and provenance can span repositories.
- req-003: Repo-relative paths remain data attributes and scopes, not primary identity.
- req-010: Migration introduces stable IDs and ownership metadata before full canonical cutover.

## Notes


## Verification
Implemented stable global ID and ownership metadata primitives in `@pi-loom/pi-storage`. Added deterministic ID helpers for spaces, repositories, worktrees, entities, links, projections, and events; repository/worktree identity resolution that prefers git remote identity over checkout path; and a SQLite catalog schema that stores `spaceId`, `owningRepositoryId`, repo-relative path scopes, versions, and projection metadata without using path-derived primary keys. Verified with `npx biome check packages/pi-storage`, `npm test -- --run packages/pi-storage/__tests__/contract.test.ts packages/pi-storage/__tests__/catalog.test.ts`, and `npm run typecheck`.
