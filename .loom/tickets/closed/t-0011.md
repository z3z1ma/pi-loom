## Summary
The repository lacks explicit rules for which `.loom` artifacts are canonical and should be committed versus which are generated/runtime outputs that should remain untracked.

## Context
Evidence:
- README and package READMEs describe `.loom` as repo-visible durable state, but do not distinguish canonical state from generated rollups or runtime handoff files.
- `.gitignore` only excludes `/.pi/loom/state.json` and gives no guidance for `.loom/**`.
- Ralph, critique, docs, plans, and specs all generate artifacts like `dashboard.json`, `launch.json`, `iterations.jsonl`, and `ticket-projection.json` whose commit-worthiness is unclear.
- Users are already seeing git noise from regenerated dashboard metadata.

## Plan
1. Classify each artifact type (`state.json`, markdown summaries, JSONL logs, dashboards, launch descriptors, projections, packets) as canonical, derived, or runtime-only.
2. Document the policy in the root README and relevant package READMEs.
3. Align `.gitignore` and/or store behavior with that policy.
4. Add tests or lint-style checks for any files that must stay generated-only or stable-on-commit.

## Notes


## Verification
A maintainer should be able to tell from repository docs and ignore rules exactly which `.loom` files belong in commits, and a clean no-op workflow should not introduce ambiguous generated files.

Documented `.loom` commit policy in README/`pi-ralph` README and enforced runtime-local launch descriptor ignores in `.gitignore`. Verified docs changes with `npm run lint` and full workspace tests/typecheck.
