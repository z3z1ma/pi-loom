## Summary
Implement the out-of-band manager supervision policy that evaluates compact worker telemetry and recent durable deltas, distinguishes busy versus idle phases, persists interventions, and enforces anti-stagnation behavior.

## Context
Why this exists: this ticket turns the strongest lesson from `pi-supervisor` into the manager-worker substrate’s actual control loop. Without it, the system can create workers and observe them but cannot meaningfully keep them unstuck or stop them from lazily drifting. This ticket should encode the rules by which the manager watches, intervenes, escalates, and avoids both over-interrupting productive workers and abandoning idle ones.

What a newcomer should understand before touching code:
- The manager must supervise from compact worker state plus recent deltas, not full transcript replay.
- Busy workers and idle/blocked workers are handled differently. The whole point is to avoid gratuitous interruptions while still ensuring that idle workers never languish unattended.
- Interventions are durable state, not ephemeral chat nudges.
- Anti-stagnation is first-class: repeated identical blockers, repeated no-progress checkpoints, or repeated ineffective interventions should trigger escalation/reassignment/retirement rather than endless steering.

Likely files and areas touched:
- supervision policy/engine code in the worker package
- intervention record models/store helpers
- dashboard/packet views that surface intervention history and stagnation state
- tests for busy-vs-idle policy, intervention persistence, and anti-loop behavior

Important edge cases:
- worker is actively editing/testing and should not be interrupted for every heartbeat
- worker is idle after asking for approval and must receive a prompt manager response
- worker keeps reporting the same blocker across several checkpoints
- worker telemetry stops advancing entirely after a runtime crash or hang
- manager analysis/runtime fails and must record that failure truthfully rather than silently pretending nothing happened

Dependencies and sequencing:
- Depends on messaging and checkpoint/telemetry tickets.
- Unblocks approval flows and final verification because those flows rely on trustworthy manager decisions.

## Plan
1. Define intervention and supervision-history records as durable state.
2. Implement phase-aware supervision rules that separate busy execution from idle, blocked, waiting, and completion-requesting phases.
3. Build anti-stagnation heuristics using checkpoint and message history rather than transcript scraping.
4. Persist manager interventions and any escalation/reassignment decisions.
5. Surface supervision state in dashboards/packets so later operators can audit why a worker was redirected or escalated.
6. Add tests that prove productive workers are not spammed while idle or blocked workers receive deterministic handling.
