## Summary
Replace direct file-backed canonical reads/writes in `pi-constitution`, `pi-research`, and `pi-initiatives` with the shared storage contract while preserving repo-materialized constitution markdown bodies and any deliberate projections.

## Context
Follow-up to the foundational storage substrate slice (`t-0044` through `t-0049`). Verification showed the migration is not complete because package stores still reopen neighboring `.loom/...` files directly. This ticket begins package-level cutover at the upstream layers so later packages can depend on shared storage truth instead of repo-file canonical state.

In scope:
- `packages/pi-constitution/`
- `packages/pi-research/`
- `packages/pi-initiatives/`
- integration with `@pi-loom/pi-storage`

Important carve-out:
- keep the constitutional brief and main constitutional markdown bodies repo-materialized and grep-friendly
- machine metadata/state moves to shared storage unless intentionally projected
- repo-relative paths remain projection/path-scope data, not identity

## Plan
1. Introduce package-local storage adapters on top of `@pi-loom/pi-storage` for constitution/research/initiative records.
2. Preserve existing markdown body rendering/materialization for constitution and any intentional review surfaces.
3. Replace direct neighboring `.loom` reads/writes used as canonical truth with shared storage reads/writes.
4. Update tests to prove canonical state now comes from shared storage while markdown projections remain available where intended.
5. Keep migration truthful: do not leave parallel canonical paths in place once each package is cut over.
