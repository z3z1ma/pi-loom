## Summary
Direct SDK worker sessions for worker launch prompts can end with an assistant message whose content array is empty, leaving the worker with no tool actions, no text output, and no durable progress. The launch path now fails truthfully, but the underlying no-op behavior still blocks real worker execution.

## Context
Observed while debugging `workspace-package-audit-worker` for audit ticket `t-0040` after fixing the false-success bug in `t-0041`.

Concrete evidence from the current main workspace:
- Running the updated repo runtime against `workspace-package-audit-worker` with `runtime: "sdk"` now blocks truthfully with `Launch reported completed but made no durable progress...`
- A direct SDK session probe using `createAgentSession({ cwd: worker.workspacePath, sessionManager: SessionManager.inMemory() })` followed by `session.prompt(renderWorkerLaunchPrompt(worker))` ended with:
  - `lastRole: "assistant"`
  - `lastContent: []`
  - `messageCount: 2`
- No tool calls, checkpoints, inbox acknowledgements, or worker-to-manager updates were recorded

This suggests the remaining issue is below the launch-status layer: the SDK session is returning an empty assistant turn for this worker prompt instead of taking action or producing an error.

Relevant code:
- `packages/pi-workers/extensions/domain/runtime.ts` (`runSdkWorkerLaunch`)
- `packages/pi-workers/extensions/domain/render.ts` (`renderWorkerLaunchPrompt`)
- potentially the underlying `@mariozechner/pi-coding-agent` SDK session behavior when launched this way

## Plan
1. Reproduce the empty-assistant behavior in a focused runtime test or harness probe.
2. Determine whether the empty response is caused by session/model setup, prompt construction, extension/resource loading, or event/result handling.
3. Change the worker SDK launch path or surrounding session setup so launches either perform work, produce actionable output, or fail with a concrete error instead of an empty assistant turn.
4. Add regression coverage around the identified root cause.
5. Re-run `workspace-package-audit-worker` to confirm a real worker turn can begin.

## Notes


## Verification
Root cause fixed in `packages/pi-workers/extensions/domain/runtime.ts` by inheriting the parent ledger/extension roots and explicitly binding built-in coding tools to the worker workspace for SDK launches. Regressions: `packages/pi-workers/__tests__/runtime.test.ts`, `packages/pi-workers/__tests__/tools.test.ts`. Verified with `npm test -- --run packages/pi-workers/__tests__/runtime.test.ts packages/pi-workers/__tests__/tools.test.ts`, `npm run typecheck`, and a live `worker_resume` of `workspace-package-audit-worker` that produced durable worker/ticket progress instead of an empty/no-op SDK turn.

Accepted and closed after review. Verified by focused runtime/tool tests (`npm test -- --run packages/pi-workers/__tests__/runtime.test.ts packages/pi-workers/__tests__/tools.test.ts`), `npm run typecheck`, and live durable worker execution proving the repo-backed SDK worker path now begins a real turn instead of ending in an empty/no-op assistant message.
