## Summary
Add the structured completion-request flow in which workers submit explicit evidence and managers record approve, reject-for-revision, or escalate decisions before any consolidation can begin.

## Context
Why this exists: the manager-worker model breaks if workers can merely say 'done' and be treated as complete. This ticket formalizes the checkpoint where a worker presents evidence and the manager makes an accountable decision. It is the decision boundary between parallelizable worker execution and the non-parallelizable approval/fan-in stage.

What a newcomer should understand before touching code:
- Workers do not self-approve.
- Completion requires a structured payload: claimed scope complete, validation evidence, remaining risks, and current branch/workspace state.
- Manager decisions must be explicit and durable: approve, reject for revision, or escalate.
- Approval is not consolidation; it only grants permission to enter the consolidation flow handled later.

Likely files and areas touched:
- worker models for completion request and approval state
- store helpers and tool/command handlers for requesting and deciding completion
- worker render/dashboard views that surface pending approvals and decision rationale
- tests around lifecycle transitions and missing evidence handling

Important edge cases:
- worker requests completion without sufficient evidence
- manager rejects with actionable rationale and the worker must return to active work
- worker becomes stale while waiting for approval
- approval decisions must remain visible after process turnover and should not require transcript reconstruction

Dependencies and sequencing:
- Depends on checkpoints/telemetry and supervision tickets because the manager needs durable evidence and policy context.
- Unblocks consolidation because only approved workers may enter fan-in.

## Plan
1. Define the structured completion request payload and approval decision model.
2. Implement lifecycle transitions into `completion_requested`, `approved_for_consolidation`, rejected/revision-needed states, and escalation states as needed.
3. Add operator/agent surfaces for workers to request completion and managers to decide it.
4. Render pending-approval and decision state clearly in worker summaries/dashboards.
5. Add tests for approval success, evidence insufficiency, rejection-for-revision, and durable decision persistence.
