## Summary
Run a fresh manager-led audit across every shipped Pi Loom package to identify overt correctness defects, brittle edge cases, and robustness gaps, then leave durable evidence and follow-up execution breadcrumbs instead of ad hoc chat notes.

## Context
This is a follow-up package scrub requested by the user after the prior workspace-wide reliability pass completed. The worker should treat the previous initiative `workspace-package-reliability-scrub` and research record `workspace-package-robustness-scrub-2026-03` as baseline context, not as proof that the codebase is now defect-free. The target surface is all ten shipped packages under `packages/`: `pi-constitution`, `pi-research`, `pi-initiatives`, `pi-specs`, `pi-plans`, `pi-ticketing`, `pi-workers`, `pi-critique`, `pi-ralph`, and `pi-docs`.

The goal is investigation first: find real bugs, robustness gaps, unsafe assumptions, stale invariants, or obvious cohesion problems that can mislead later maintainers or break behavior outside the happy path. The worker should reason from callers and contracts outward, inspect tool/command surfaces, stores, dashboards, packets, runtime/state transitions, normalization, rendering, and tests, and prefer concrete evidence over vague suspicion.

This ticket is not asking for speculative architecture work or broad feature invention. If the worker finds a small, obviously correct fix with tight scope and clear verification, it may implement it in its worker workspace and record the proof. For anything larger, riskier, or ambiguous, it should capture a precise finding and create or recommend follow-up tickets instead of bluffing through a partial fix.

Constraints:
- Keep Loom layer boundaries truthful; do not solve review findings by adding compatibility shims or duplicate representations.
- Prefer durable execution evidence: worker checkpoints, ticket notes, created follow-up tickets, and concrete validation commands.
- Distinguish confirmed bugs from hypotheses. A plausible concern without repo evidence is not a finished finding.
- Avoid rerunning the exact previous scrub mechanically; use the prior initiative/research for orientation, then inspect current code and tests directly.
- Before changing exported APIs or shared contracts, follow references and account for all consumers.
- If fixes are made, add targeted regression coverage where practical and record exact verification results.

## Plan
1. Build a package-by-package audit map covering all ten shipped packages and the most defect-prone surfaces in each package.
2. Inspect current package entrypoints, domain stores, tools/commands, render/normalize helpers, runtime/state transitions, and targeted tests to look for overt correctness and robustness defects.
3. Record concrete findings with file-level evidence, affected contracts, failure mode, and recommended action.
4. Apply only tightly scoped, high-confidence fixes inside the worker workspace when they are clearly correct and cheaply verifiable; otherwise create or recommend follow-up tickets.
5. Leave a durable worker checkpoint and ticket notes summarizing coverage, findings, any fixes made, follow-up tickets created, and remaining risk.
6. If code changes land in the worker workspace, run the narrowest meaningful verification for each touched area and record exact commands/results.
