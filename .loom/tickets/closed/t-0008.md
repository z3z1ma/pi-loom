## Summary
Several packages persist cwd-derived absolute paths into `.loom` artifacts, making checked-in state machine-specific and non-portable across clones.

## Context
Evidence:
- `packages/pi-initiatives/extensions/domain/dashboard.ts` writes `linkedResearch.items[].path` using `join(cwd, ".loom", "research", researchId)`.
- `packages/pi-plans/extensions/domain/store.ts` passes absolute `planDir`, packet, and markdown paths into `buildPlanDashboard()`, which persists them in `dashboard.json`.
- `packages/pi-critique/extensions/domain/store.ts` persists absolute `packetPath`/`launchPath` in both `dashboard.json` and `launch.json`.
- `packages/pi-docs/extensions/domain/store.ts` and `dashboard.ts` persist absolute doc, packet, and dashboard paths.
- `packages/pi-ralph/extensions/domain/dashboard.ts` and `store.ts` persist absolute run, packet, and launch paths.
Checked-in examples under `.loom/` already show absolute paths, e.g. critique launch descriptors.

## Plan
1. Inventory every persisted path-bearing field across Loom layers.
2. Convert durable artifacts to repo-relative paths or stable ids/refs resolvable from the current workspace root.
3. Update readers/renderers/runtime consumers to resolve relative paths at use time.
4. Add regression tests that assert persisted artifacts never contain the current absolute workspace root.
5. Migrate checked-in sample `.loom` artifacts to the normalized representation.

## Notes


## Verification
Create/update/read flows for the affected packages should produce durable artifacts with no absolute workspace prefix, and relocation of the repo should not invalidate those artifacts.

Implemented repo-relative path storage/output across the affected Loom layers. Verified with `npm run typecheck`, `npm run lint`, and `npm test`.
