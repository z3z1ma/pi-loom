## Summary
Define the canonical worker domain model, normalization rules, and rendered state so a worker has stable identity, explicit linked Loom refs, durable lifecycle fields, and portable workspace intent without leaking clone-local runtime paths into canonical state.

## Context
Why this exists: once the package scaffold exists, the next hard problem is saying exactly what a worker record knows. This ticket turns the abstract spec into concrete domain types and rendering rules. If this model is wrong, every later feature will grow around a lie. The key design pressure is portability: workers are backed by ephemeral workspaces, but committed Loom truth must remain portable across machines and clones.

What a newcomer should understand before touching code:
- The worker record is the durable anchor; runtime attachments are secondary and machine-local.
- Store enough information to understand the assignment, relationships, and current state without needing the live workspace path.
- Every worker must link to at least one ticket and may additionally link to plans, specs, initiatives, research, critiques, Ralph runs, and docs.
- The workspace descriptor should capture intent (`repo root`, `base ref`, branch strategy, workspace strategy, logical labels), not clone-local absolute filesystem facts.
- This ticket should produce the types and renders that later tickets rely on for messaging, telemetry, supervision, and recovery.

Likely files and areas touched:
- `packages/pi-workers/extensions/domain/models.ts`
- `packages/pi-workers/extensions/domain/normalize.ts`
- `packages/pi-workers/extensions/domain/store.ts`
- `packages/pi-workers/extensions/domain/render.ts`
- `packages/pi-workers/extensions/domain/dashboard.ts` if the package mirrors neighboring layers
- tests around model normalization, portability, and markdown rendering

Important fields to think through carefully:
- worker id, title/objective, manager source ref, linked ticket ids, optional linked Loom refs
- lifecycle status and timestamps
- target branch / consolidation target metadata
- portable workspace descriptor
- current checkpoint summary and latest telemetry summary placeholders
- completion request / approval / consolidation placeholders even if richer behavior lands later

Non-goals:
- do not implement actual worktree provisioning yet
- do not implement message-stream writes yet except whatever minimal placeholders the model requires
- do not hide ambiguity by storing absolute paths in canonical state

Dependencies and sequencing:
- Depends on the package scaffold ticket.
- Unblocks runtime attachment, provenance integration, and many later tickets, so preserve strong type boundaries and clear render semantics.

## Plan
1. Define the worker state model and any supporting enums/unions for lifecycle and manager references.
2. Define a portable workspace descriptor shape that can represent intent without machine-local attachment facts.
3. Implement normalization rules that reject or strip clone-local absolute path leakage from canonical state.
4. Update store code so worker create/read/update flows preserve the new model correctly.
5. Render `worker.md` and any summary/dashboard projections so operators can inspect worker state without reading raw JSON.
6. Add tests covering normalization, persistence, render output, and linked-ref fidelity.
7. Ensure the resulting model leaves room for follow-on messaging, telemetry, and approval tickets without forcing a breaking redesign.
