## Summary
Change worker run semantics so each run consumes unresolved inbox state, records message-handling progress durably, re-checks the inbox before stopping, and exits only at explicit bounded stop conditions.

## Context
Why this ticket exists: once the inbox protocol is explicit, worker execution needs to actually honor it. Right now workers are still too close to one-shot resumable subprocess turns. This ticket makes worker runs meaningfully inbox-driven without yet requiring a permanent daemon model.

What a newcomer should understand:
- a worker run is no longer just 'run prompt once'
- before substantive work, the worker must inspect unresolved manager messages and relevant broadcast backlog
- during the run, it must acknowledge, resolve, or escalate those messages as appropriate
- before stopping, it must re-check whether new unresolved inbox work arrived
- legitimate stop conditions must be explicit and bounded

Likely files / areas:
- `packages/pi-workers/extensions/domain/store.ts`
- `packages/pi-workers/extensions/domain/runtime.ts`
- `packages/pi-workers/extensions/domain/render.ts`
- `packages/pi-workers/extensions/tools/worker.ts`
- `packages/pi-workers/__tests__/runtime.test.ts`
- `packages/pi-workers/__tests__/store.test.ts`

Important constraints:
- still preserve bounded worker turns; this ticket does not require an always-on daemon
- do not silently drop unresolved manager work at the end of a run
- checkpoints must start reflecting inbox-processing progress, not just implementation progress
- stop conditions must be explicit: inbox empty, blocked on manager, awaiting approval/review, or bounded policy budget reached

This ticket depends on the inbox protocol ticket because worker turns need a real inbox contract to act on.

## Plan
1. Define the worker-turn contract around unresolved inbox state and bounded stop conditions.
2. Update launch/resume execution flow so runs load unresolved inbox work before starting.
3. Ensure runs record acknowledgment/resolution/escalation outcomes as they process messages.
4. Re-check unresolved inbox state before stopping and preserve durable evidence if work remains.
5. Extend checkpoint semantics/rendering to include inbox-handling progress.
6. Add tests that prove workers drain inbox state until a valid stop condition rather than exiting arbitrarily.
