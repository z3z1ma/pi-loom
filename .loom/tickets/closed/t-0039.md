## Summary
Close the control-plane phase with comprehensive tests, prompt guidance, README updates, and neighboring package integration checks so the stronger inbox protocol, manager surface, runtime abstraction, and scheduler behavior are all documented and defended by automation.

## Context
Why this ticket exists: this phase changes not just internals but how the system is supposed to be used. If docs, guidance, and tests lag behind, the control plane will immediately become confusing again. This ticket locks in the new behavior as product truth.

What a newcomer should understand:
- this ticket is the contract hardening phase, not optional polish
- prompt guidance is part of product behavior in this repo
- docs must explain what changed without overstating actor-mesh capabilities or pretending humans are already completely removed from the loop
- neighboring integrations must remain truthful about ticket primacy, worker semantics, and Ralph boundaries

Likely files / areas:
- `packages/pi-workers/README.md`
- worker prompt guidance files/tests
- manager/worker command/tool/store/runtime tests
- any touched neighboring package docs/tests where integrations changed
- root docs if public stack descriptions or usage guidance need updating

Important constraints:
- keep the bounded architecture explicit
- do not let docs imply unrestricted real-time actor mesh if that still is not implemented
- preserve ticket/Ralph/worker boundary truth

This ticket depends on the main semantic/runtime/scheduler work because it should document and test the finished integrated phase, not placeholders.

## Plan
1. Audit the integrated control-plane changes and ensure every capability has direct automated coverage.

3. Update package README and prompt guidance to teach the new phase correctly.
4. Update neighboring package integrations/docs/tests as needed so ticket primacy and Ralph boundedness remain clear.
5. Capture recommended verification commands for future maintainers.
