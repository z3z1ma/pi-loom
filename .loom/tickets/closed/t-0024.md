## Summary
Provide the first-class operator and agent surface for the worker substrate through `/worker` commands, `worker_*` tools, and prompt guidance that teaches the right architecture and keeps the feature usable in both interactive and headless environments.

## Context
Why this exists: even a correct worker substrate is not usable if operators and agents cannot interact with it through a clean first-class interface. This ticket exposes that surface and encodes the doctrine that prevents misuse. It is also where the package starts teaching the broader agent ecosystem how to think about workers truthfully.

What a newcomer should understand before touching code:
- The command/tool surface must cover the full lifecycle that operators need: create, read, list, message, checkpoint, approve, reject, launch, resume, retire, inspect.
- The surface must work in headless contexts; UI components may help, but the canonical path is through durable state and tool results.
- Prompt guidance matters in this repo. It is part of product behavior and needs tests.
- This ticket should teach boundaries clearly: workers are workspace-backed units, managers supervise out-of-band, tickets remain execution truth, and Ralph stays bounded.

Likely files and areas touched:
- `packages/pi-workers/extensions/commands/*.ts`
- `packages/pi-workers/extensions/tools/*.ts`
- `packages/pi-workers/extensions/prompts/guidance.ts` and prompt base markdown
- command/tool registration in `extensions/index.ts`
- tests for command/tool registration and guidance text

Important edge cases:
- headless mode where UI dialogs/widgets are unavailable
- operators needing enough information from tool output to proceed without opening files manually
- guidance accidentally encouraging session-branch or generic-subagent misuse
- naming and affordances that should remain consistent with existing Loom packages

Dependencies and sequencing:
- Depends on runtime attachment, messaging, telemetry, and approval flows so the surface can expose real behaviors rather than stubs.
- Feeds recovery/retirement and final documentation/testing work.

## Plan
1. Define the `/worker` command namespace and `worker_*` tool set needed for the v1 lifecycle.
2. Implement command/tool handlers over the underlying store/runtime/messaging/checkpoint/approval APIs.
3. Add prompt guidance and package README language that teaches correct use and layer boundaries.
4. Ensure headless flows return structured information sufficient for operation without interactive UI.
5. Add registration and prompt-guidance tests proving the surface exists and encodes the right doctrine.
