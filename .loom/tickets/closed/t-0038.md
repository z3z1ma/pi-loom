## Summary
Expand worker dashboards, packets, queue summaries, and recovery behavior so unresolved inbox state, runtime kind, scheduler visibility, and restart-safe reconstruction remain obvious across the stronger control plane.

## Context
Why this ticket exists: richer inbox semantics, live runtimes, and manager scheduling all increase the amount of state the system must explain. Without stronger observability and recovery, the control plane will become harder to trust exactly when it becomes more autonomous. This ticket keeps the system understandable.

What a newcomer should understand:
- recovery means reconstructing truth from durable records after interruption, not trusting in-memory scheduler/session state
- observability must now include unresolved inbox counts, acknowledgment lag, runtime kind, and scheduler-visible state in addition to existing telemetry/checkpoints
- canonical worker state must remain portable even while richer runtime metadata is added

Likely files / areas:
- worker dashboard and packet rendering
- worker store/recovery helpers
- runtime descriptors and read surfaces
- tests around restart-safe reconstruction and path portability

Important constraints:
- do not leak machine-local runtime details into canonical committed worker state
- do not assume one runtime kind; recovery must work across runtime abstractions
- dashboards should stay useful to both humans and AI operators without transcript archaeology

This ticket depends on the inbox protocol, runtime abstraction, and scheduler work because it needs real control-plane state to surface.

## Plan
1. Extend dashboards/packets to surface inbox backlog, acknowledgment lag, runtime kind, scheduler visibility, and review backlog.
2. Update recovery paths so replacement managers can rebuild queue state from durable worker state and runtime descriptors.
3. Preserve path portability and keep machine-local runtime details out of canonical state.
4. Add tests proving restart-safe reconstruction and non-leakage of runtime-specific machine-local data.

## Notes
Understandability ticket. More automation means stronger observability and recovery must follow.
