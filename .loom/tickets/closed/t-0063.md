## Summary
Address post-rollout problems in the new `/ticket` overlay workbench: broken arrow/Esc behavior, missing overflow management, weak fixed-size layout, and insufficient styling compared with the PI settings and pi-subagents references.

## Context
The first ticket workbench rollout replaced the old action-list UI, but real usage exposed important interaction and polish problems: scrolling and overflow feel broken, overlay sizing is not convincingly bounded, arrow keys and Esc do not behave reliably, and the visual treatment is still far below the PI settings/pi-subagents references that motivated the redesign. This is not a cosmetic-only request; the shell is still untrustworthy if navigation and dismissal are flaky.

The fix should stay within `pi-ticketing` unless the root cause is undeniably a shared pi-tui or runtime issue. Start by confirming how reference shells in local PI resources handle focus, keyboard routing, and bounded overlays, then bring the ticket workbench up to that standard.

## Plan
1. Inspect the current ticket workbench implementation and compare it against local reference shells and pi-tui components that already solve scrolling, selection, and overlay layout.
2. Redesign the workbench to use stronger interactive primitives for keyboard navigation, visible focus, and overflow handling instead of relying on raw rendered strings alone.
3. Improve styling and hierarchy so the shell feels intentionally designed: stronger tabs, cards/panels, clearer focus, and tasteful expressive touches aligned with the user's request.
4. Add regression coverage for keyboard handling and any new scrolling/focus behaviors that can be tested headlessly.
5. Run targeted ticketing verification and a critique cycle after the refinement lands.
