## Summary
Rewrite constitutional and README policy so canonical DB state, repo-materialized main constitution/docs/spec markdown carve-outs, and projection boundaries are explicit, then define the backend-agnostic storage contract for entities, links, events, projections, projects, repositories, and worktrees.

## Context
Projected from finalized spec `sqlite-first-canonical-storage-substrate`. This ticket must make the storage policy truthful before implementation. The critical carve-out is not that all constitution/docs/spec artifacts remain in git. It is that the main human-facing markdown bodies remain repo-present and grep-friendly while machine-oriented metadata/state moves to the canonical database by default. For `pi-constitution`, keep the constitutional brief and main constitutional markdown bodies in-repo. For `pi-docs`, keep the main doc markdown body/bodies in-repo. For `pi-specs`, keep the main spec markdown bodies (`proposal.md`, `design.md`, `tasks.md`, and related human-facing spec text) in-repo. JSON metadata, dashboards, machine state, and similar auxiliary artifacts should move to the DB unless there is a specific projection reason to materialize them.

## Plan
1. Update constitutional and README policy so operational Loom truth is DB-canonical rather than path-derived repo files.
2. Define the backend-agnostic storage contract for spaces/projects, repositories, worktrees, entities, links, events, projections, and repo-relative path scopes.
3. Explicitly document the repo carve-out: main constitution/docs/spec markdown bodies remain grep-friendly in git; metadata/state does not remain committed by default.
4. Identify package boundaries that currently reopen neighboring `.loom` files directly and map them onto the new contract.
