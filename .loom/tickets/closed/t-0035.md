## Summary
Add a same-process SDK-backed worker host that can execute inbox-driven worker turns with direct Pi session control while persisting all important state transitions back into the durable worker ledger.

## Context
Why this ticket exists: the new runtime research says SDK is the cleanest same-runtime control surface for live workers. This ticket turns that research into a concrete runtime path. The goal is not to bypass the worker ledger; the goal is to host live worker execution in a stronger runtime without making worker truth disappear into process memory.

What a newcomer should understand:
- SDK is the preferred next runtime because it offers direct session control and event subscriptions
- the worker ledger remains canonical even when execution becomes more interactive/live
- this ticket should build on the runtime abstraction, not around it
- parity with the existing worker contract matters more than fancy live-runtime behavior

Likely files / areas:
- worker runtime abstraction implementation files
- SDK integration code using Pi session APIs
- worker launch/resume paths that select SDK runtime
- tests or fixtures proving durable state parity

Important constraints:
- subprocess runtime stays available as baseline/compatibility path
- SDK runtime must persist message, checkpoint, telemetry, approval, and recovery semantics back into durable state
- do not create a hidden in-memory control plane that only exists while the host process is alive

This ticket depends on the runtime abstraction ticket. It should land before the manager scheduler, because the scheduler needs a stronger live-worker path to drive.

## Plan
1. Implement the SDK-backed runtime behind the new runtime abstraction.
2. Ensure worker launch/resume can select or use the SDK runtime path.
3. Persist all important session-driven state changes back into worker records and runtime descriptors.
4. Compare behavior against subprocess execution for overlapping worker contract semantics.
5. Add tests or equivalent validation proving durable parity and truthful recovery.

## Notes
Preferred live-runtime ticket. Treat the SDK as an execution surface, not as the source of worker semantics.
