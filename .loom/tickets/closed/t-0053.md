## Summary
Rewire `pi-workers` so durable worker history uses the shared storage contract while clone-local runtime attachments remain local, then cut worker/manager tools over to the new canonical path.

## Context
Follow-up to the foundational storage substrate slice (`t-0044` through `t-0049`). The storage package now distinguishes canonical worker history from clone-local runtime attachments, but `pi-workers` itself still uses the old file-backed store and tool path. This ticket performs the actual package cutover.

In scope:
- `packages/pi-workers/`
- integration with `@pi-loom/pi-storage`
- worker/manager command and tool surfaces

Important boundary:
- durable worker messages/checkpoints/approvals/history use shared storage
- clone-local runtime attachments, launch descriptors, leases/heartbeats, and worktree/process state remain local-only
- package behavior must not regress manager/worker semantics while changing the storage substrate

## Plan
1. Add worker storage adapters on top of `@pi-loom/pi-storage` for durable worker history.
2. Keep local runtime attachment handling on the local runtime store instead of the canonical SQLite catalog.
3. Rewire worker/manager tools and commands to use the new storage-backed history and projections.
4. Update focused tests to prove canonical-vs-local separation, durable worker history portability, and unchanged manager/worker semantics where intended.
