## Summary
Implement deterministic recovery, resume, reprovision, and retirement behavior so managers can safely recover from crashes, stale attachments, or abandoned workers without relying on the original live process or transcript.

## Context
Why this exists: long-horizon systems fail in boring ways—processes crash, paths disappear, terminals close, and worktrees get cleaned up. This ticket makes the worker substrate resilient to those realities. It should ensure that durable state is actually enough to continue or retire work cleanly, which is one of the main reasons the substrate exists at all.

What a newcomer should understand before touching code:
- Recovery is not just 'restart the process.' The system must inspect durable state, determine whether the runtime attachment is still valid, and either resume, reprovision, or retire the worker explicitly.
- Retirement is a first-class outcome, not an error path. It should preserve history and explain why the worker is no longer active.
- Recovery behavior must not depend on interactive UI or hidden in-memory session state.
- The package should remain truthful even when the previous machine-local worktree path no longer exists.

Likely files and areas touched:
- runtime/recovery helpers
- lifecycle transition and retirement logic in the store
- command/tool handlers for resume/retire/recover operations
- tests around stale runtime descriptors, missing paths, and clean retirement history

Important edge cases:
- runtime descriptor exists but process is gone
- worktree path no longer exists and must be reprovisioned
- worker is in completion_requested or approved_for_consolidation when interruption happens
- worker should be retired intentionally rather than resumed
- cleanup should not erase durable audit history

Dependencies and sequencing:
- Depends on runtime attachment, telemetry, and operator surface tickets.
- Should land before the final verification/documentation ticket so recovery guarantees are testable and documented.

## Plan
1. Define explicit recovery and retirement decision paths based on durable worker state and runtime descriptor validity.
2. Implement resume/reprovision helpers for stale or missing attachments.
3. Implement retirement flows that preserve durable history while marking the worker no longer active.
4. Surface recovery/retirement state clearly in worker summaries and dashboards.
5. Add tests for interrupted workers, missing worktrees, stale descriptors, successful resume, and intentional retirement.

## Notes
Resilience ticket. Durable state only matters if recovery and retirement are trustworthy.
