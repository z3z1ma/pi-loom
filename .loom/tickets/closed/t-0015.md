## Summary
Create the first truthful home for the manager-worker substrate by adding a new `packages/pi-workers/` extension package, wiring it into the workspace, and bootstrapping the durable `.loom/workers/` ledger shape. This is the foundation ticket: every later worker capability depends on the package, ledger initialization, prompt guidance surface, and repo-level integration created here.

## Context
Why this exists: the finalized spec explicitly rejects session branches, session forks, and generic task subprocesses as truthful worker representations. Pi Loom needs a first-class package whose public doctrine, ledger paths, and extension surface say what the system now is: a worker is a workspace-backed execution unit with durable state. Until this package exists, every downstream implementation would be forced to scatter worker concepts across unrelated packages or improvise on Ralph/task/session semantics.

What a newcomer should understand before touching code:
- Follow the standard package pattern already used across the repo: `README.md`, `extensions/index.ts`, `extensions/commands/`, `extensions/tools/`, `extensions/domain/`, `extensions/prompts/`, and package tests under `packages/pi-workers/__tests__/`.
- The ledger home should be `.loom/workers/`, analogous to how Ralph uses `.loom/ralph/` and plans use `.loom/plans/`.
- Canonical durable artifacts should include `state.json`, `worker.md`, `messages.jsonl`, `checkpoints.jsonl`, and `dashboard.json`. A `launch.json` or comparable runtime descriptor may exist, but it is runtime-only and must not become committed canonical truth.
- This ticket should not try to fully model runtime attachment, messaging semantics, or supervision behavior. Its job is to create the package, the ledger bootstrap, the base domain types/store skeleton, and the prompt/README language that lock in the truthful concept.

Likely files and areas touched:
- root `package.json` and `README.md` so the new package is discoverable and documented
- `packages/pi-workers/README.md`
- `packages/pi-workers/extensions/index.ts`
- `packages/pi-workers/extensions/domain/{paths,models,store,render,normalize}.ts` or equivalent starter set
- `packages/pi-workers/extensions/prompts/{guidance.ts,base-*.md}`
- initial tests such as `index.test.ts`, `store.test.ts`, and `prompt-guidance.test.ts`

Non-goals for this ticket:
- do not implement Git worktree provisioning yet
- do not implement the full message bus or supervisor engine yet
- do not wire every neighboring package yet beyond the minimal package registration and ledger existence needed for future tickets

Dependencies and sequencing:
- This ticket intentionally has no dependencies and must land before all other worker tickets.
- Downstream tickets will rely on the naming, ledger paths, and package conventions established here, so keep those decisions clear and minimal rather than speculative.

## Plan
1. Add the new `packages/pi-workers/` package following the established workspace conventions used by existing Loom packages.
2. Wire the package into the root workspace integration points so it is treated as a first-class extension package.
3. Add minimal domain path/model/store scaffolding sufficient to initialize `.loom/workers/` and create/read a skeletal worker record.
4. Define the canonical artifact names and ensure the runtime-only descriptor is clearly separated from committed state.
5. Add initial prompt guidance and README language that teach the exact boundary: worker = workspace-backed execution unit, not branch/fork/subagent.
6. Add initial tests proving ledger initialization, basic record scaffold creation, and prompt guidance wording.
7. Keep implementation intentionally thin; leave rich behavior for follow-on tickets instead of guessing ahead.
