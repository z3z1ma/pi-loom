## Summary
Fresh subprocess launches use the user workspace as `pi -e` root instead of the extension package root, which can omit the required extension tooling in installed or externally-consumed setups.

## Context
Evidence:
- `packages/pi-critique/extensions/domain/runtime.ts`, `packages/pi-docs/extensions/domain/runtime.ts`, and `packages/pi-ralph/extensions/domain/runtime.ts` compute `findExtensionWorkspaceRoot(cwd)` from the caller workspace.
- Callers pass `ctx.cwd` from the user repo (`extensions/tools/*.ts`).
- The extension manifests actually live in the package roots, and package READMEs document `omp -e .` from inside each package.
This means a fresh subprocess can start with the wrong extension set whenever the user workspace does not itself declare these extensions.

## Plan
1. Resolve the extension/package root from the current module/package metadata rather than the caller workspace.
2. Share the spawn-root discovery logic across critique/docs/ralph to avoid divergence.
3. Add tests for installed-workspace and packaged-runtime scenarios.
4. Verify the subprocess sees the expected tool surface after launch.

## Notes


## Verification
Launching critique/docs/ralph subprocess helpers from a consumer workspace that does not declare these extensions should still load the correct package extension and complete successfully.

Runtime helpers now resolve extension roots from package/module location. Verified with focused runtime tests and the full workspace `npm test`, plus `npm run typecheck` and `npm run lint`.
