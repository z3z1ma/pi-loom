## Summary
Implement the append-only manager-worker message stream so instructions, escalations, help requests, and bounded broadcast signals survive runtime turnover and can be inspected as canonical coordination history.

## Context
Why this exists: the worker substrate needs a durable coordination channel that is independent of any one transcript or live process. This ticket creates that channel. It should make message flow explicit and auditable rather than leaving manager-worker communication trapped in transient sessions. The resulting message stream becomes one of the compact inputs the manager can supervise from later.

What a newcomer should understand before touching code:
- The durable message stream is per worker and append-only.
- Direction matters: `manager_to_worker`, `worker_to_manager`, and bounded `broadcast` are semantically different.
- Default routing is manager-mediated; v1 does not include unrestricted worker-to-worker chat.
- Messages should be typed and linked to related refs/status so they are actionable records, not anonymous prose blobs.
- Broadcast is only for urgent shared information and must remain visible to the manager.

Likely files and areas touched:
- worker messaging models/store helpers
- tool/command handlers for sending and reading messages
- renderers or summary/dashboard helpers that surface unresolved messages
- tests around append-only history, message direction, and manager visibility

Important design questions this ticket must answer concretely:
- what message kinds exist initially (assignment, clarification, unblock, escalation, checkpoint notice, completion request notice, broadcast warning, etc.)
- how delivery/resolution status is recorded
- how causal links to earlier messages are represented
- how related ticket/plan/spec refs are attached

Non-goals:
- do not build the full manager policy engine yet
- do not implement unrestricted peer messaging
- do not collapse messages into informal notes sections in worker markdown

Dependencies and sequencing:
- Depends on scaffold and worker model tickets.
- Unblocks checkpoint/telemetry and supervision tickets because they need a durable coordination substrate.

## Plan
1. Define the durable message record schema, including ids, direction, kind, timestamps, related refs, and status fields.
2. Implement append-only write/read/query helpers for per-worker message history.
3. Add operator/agent surfaces for sending manager-to-worker and worker-to-manager messages.
4. Add bounded broadcast semantics that preserve explicit scope and manager visibility.
5. Surface unresolved/recent messages in worker summaries or dashboards where appropriate.
6. Add tests for append-only behavior, message ordering, causal links, broadcast visibility, and restart-safe reads.
