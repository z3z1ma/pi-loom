## Summary
Several packages expose absolute filesystem paths in summaries and read results, which makes tool output machine-specific and encourages those paths to leak into downstream artifacts.

## Context
Evidence:
- `packages/pi-ticketing/extensions/domain/frontmatter.ts` stores the parsed ticket file path on `TicketRecord.path`, and `query.ts` / `graph.ts` expose it through `TicketSummary.path`.
- `packages/pi-research/extensions/domain/store.ts` returns `ResearchSummary.path` using the absolute research directory.
- `packages/pi-plans/extensions/domain/dashboard.ts`, `packages/pi-docs/extensions/domain/dashboard.ts`, and `packages/pi-ralph/extensions/domain/dashboard.ts` all model summary `path` fields that are currently absolute before some consumers relativize them.
This is broader than persisted dashboards: even plain tool output becomes host-specific and easy for other layers to serialize unchanged.

## Plan
1. Define a cross-layer path contract for tool outputs: repo-relative string, artifact ref, or explicit `{rootRelativePath}` shape.
2. Update ticketing/research/plan/docs/ralph summaries and any dependent renderers or consumers.
3. Add tests asserting tool outputs do not expose the current absolute workspace root.
4. Audit downstream stores to prevent reintroducing raw absolute paths.

## Notes


## Verification
Reading/listing Loom records from two different clone locations should produce identical path-bearing tool output after normalizing the workspace root.

Normalized surfaced path fields to repo-relative values across ticketing, research, specs, initiatives, plans, critique, docs, and Ralph. Verified with package tests plus full workspace `npm run typecheck`, `npm run lint`, and `npm test`.
