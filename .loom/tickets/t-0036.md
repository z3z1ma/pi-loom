## Summary
Add a bounded RPC-backed runtime seam or implementation so process-separated live workers remain possible without letting RPC redefine the worker contract or become the semantic center of the package.

## Context
Why this ticket exists: the new research says RPC is the strongest documented cross-process control surface, but it should be a fallback transport, not the core domain model. This ticket preserves that option without letting the package drift toward protocol-first worker semantics.

What a newcomer should understand:
- this ticket is about transport flexibility, not about changing worker meaning
- RPC is useful when stronger process isolation or external hosting is needed
- worker contract parity matters more than transport-specific cleverness
- it is acceptable for this ticket to land as a bounded seam/documented implementation path rather than a maximal production-ready transport stack

Likely files / areas:
- runtime abstraction layer
- any RPC runtime adapter or documented seam/stub
- runtime docs/README/guidance updates
- runtime-focused tests or fixtures if an implementation lands now

Important constraints:
- do not let RPC become the default semantic center of the package
- maintain parity with subprocess/SDK worker contract semantics
- preserve the bounded architecture and keep sidecar/event-mesh ambitions out of this ticket

This ticket depends on runtime abstraction but can proceed in parallel with SDK runtime work if the abstraction boundary is stable.

## Plan
1. Extend the runtime abstraction with an RPC-capable strategy seam.
2. Implement a bounded RPC runtime or an explicit fallback path/documented stub, depending on what the codebase can support cleanly now.
3. Ensure the worker contract remains transport-neutral.
4. Add documentation/tests that make RPC’s fallback role clear and verify parity where functionality overlaps.

## Notes
Fallback-runtime ticket. Keep transport subordinate to worker semantics.

## Verification

- documentation makes it clear RPC is fallback transport, not domain model
