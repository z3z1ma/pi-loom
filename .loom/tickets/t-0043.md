## Summary
Investigate and reduce the runtime cost that makes the heaviest `pi-workers` and `pi-plans` tests exceed default per-test budgets during repo-wide `npm test` runs, even though they pass in isolation.

## Context
During the repo-wide package audit in `t-0040`, a full `npm test` run failed because `packages/pi-workers/__tests__/index.test.ts` and `packages/pi-plans/__tests__/store.test.ts` timed out under the full suite while still passing when run directly. The immediate audit fix was to raise the affected test timeouts so the full suite becomes reliable again. That fix stabilizes CI-facing behavior, but it does not explain or reduce the underlying latency cost.

Concrete evidence from the audit:
- `npm test` initially failed in the worker workspace with two timeout failures while 62 other files passed.
- Running package suites individually succeeded for all ten shipped packages.
- The heaviest package durations were `pi-plans` (~6.7s file duration) and `pi-workers` (~7.8s file duration), with worker-related tests also printing repeated `Preparing worktree ...` output.

This follow-up should treat the problem as a robustness/perf issue in the test harness and/or package setup path, not as a request to weaken coverage. The goal is to understand whether the cost comes from repeated worktree setup, dynamic imports, packet compilation breadth, Vitest parallelism interactions, or another avoidable source.

## Plan
1. Reproduce full-suite timing with targeted timing instrumentation for the expensive `pi-workers` and `pi-plans` tests.
2. Identify which setup path dominates wall-clock time (e.g. worktree provisioning, extension import graph, packet compilation breadth, or cross-test contention).
3. Apply a bounded optimization or harness change that reduces runtime without shrinking the behavioral surface under test.
4. Keep or revise timeout budgets only after the runtime profile is understood and documented.
5. Verify with `npm test` and the narrow package test commands that originally reproduced the issue.
