## Summary
Introduce `/manager` and `manager_*` surfaces so a manager can supervise many workers intentionally through a native control plane instead of stitching together raw `worker_*` operations ad hoc.

## Context
Why this ticket exists: today the manager is mostly an implicit role using low-level worker tools manually. That is workable for development, but it does not give the AI manager an explicit orchestration surface. This ticket should make manager behavior concrete and operable.

What a newcomer should understand:
- manager is still a role, not a new memory layer
- the new surface should be an orchestration/control plane over durable worker state
- manager operations must remain durable and auditable
- the point is not magic autonomy; the point is a truthful surface for worker fleet management

Likely files / areas:
- `packages/pi-workers/extensions/commands/manager.ts` (or equivalent)
- `packages/pi-workers/extensions/tools/manager.ts`
- `packages/pi-workers/extensions/index.ts`
- prompt guidance and README updates inside `packages/pi-workers`
- manager-focused tests under `packages/pi-workers/__tests__/`

Key operations the new surface should cover:
- list workers and queue state
- inspect unresolved inbox and pending approvals across workers
- supervise one or many workers from compact state
- send manager-to-worker instructions
- resume eligible workers
- process or surface completion approvals

Important constraints:
- manager actions must become durable worker/ticket-visible state
- do not bypass the worker ledger with hidden command-only logic
- keep the surface headless-safe
- do not let this become a hidden workflow engine or replace worker/ticket truth

This ticket depends on the inbox protocol because manager orchestration must operate on explicit inbox state.

## Plan
1. Define the `/manager` command family and `manager_*` tool surface.
2. Implement manager-facing operations over the worker store and supervision state.
3. Ensure manager actions write durable evidence rather than hidden orchestration state.
4. Add prompt guidance and README language that explain what the manager surface does and does not do.
5. Add command/tool tests for multi-worker inspection, messaging, supervision, and approval handling.

## Notes
Control-surface ticket. This is what gives the AI manager a truthful native orchestration interface.
